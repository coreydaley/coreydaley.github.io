# Sprint 001: Co-located Page Bundles with Year/Month Organization

## Overview

The blog currently suffers from a flat-file problem: 20 posts live in `content/posts/`
and their images live in `static/images/posts/`, with no enforced naming convention
linking the two. Finding which images belong to a post, or adding images to an existing
post, requires cross-referencing filenames by memory. For AI agents, this is even worse —
there is no reliable way to infer the relationship without reading every post's frontmatter.

This sprint migrates the site to Hugo **leaf bundles** organized by year and month:

```
content/posts/2026/02/welcome-to-my-new-blog/
├── index.md                    ← the post
├── robot-writing-code.png      ← original (preserved, stripped by CI)
└── robot-writing-code.webp     ← optimized WebP
```

Every post's content and images live together in one directory. Anyone — human or AI —
can navigate to a post directory and find everything related to that post in one place.
Old URLs (`/posts/welcome-to-my-new-blog/`) redirect to new URLs
(`/posts/2026/02/welcome-to-my-new-blog/`) via Hugo `aliases`, which generate static
HTML redirect pages compatible with GitHub Pages.

Image references in `figure-float`/`figure-block` shortcodes shift from absolute
`/images/posts/filename.webp` paths to page-resource-relative paths (just `filename.webp`),
which Hugo resolves automatically within a leaf bundle. The `image` frontmatter field
(used for og:image) shifts to a relative path as well — the theme template's `<meta>`
tag will need a one-line update to resolve bundle resources.

## Use Cases

1. **Author adds images to a post**: Drop the PNG into the post's bundle directory,
   run `./scripts/optimize-images.sh`, use the filename in the shortcode. No hunting
   through a shared image pool.

2. **AI agent creates a new post**: Run `hugo new posts/2026/03/my-topic/index.md`,
   place generated image in `content/posts/2026/03/my-topic/`, run image optimization.
   The agent can verify co-location with a single `ls` on the post directory.

3. **Reader follows an old bookmarked URL**: `/posts/old-slug/` serves a static
   HTML redirect page (generated by Hugo from `aliases`) that redirects to the new URL.

4. **Developer browses the repo on GitHub**: The directory tree immediately communicates
   post structure and which images belong to which post, without opening any file.

5. **Search/Pagefind indexing**: Unchanged — Hugo builds the same HTML, Pagefind indexes
   from the `public/` output, URLs in search results will reflect the new paths.

### URL Changes

| Old URL | New URL | Redirect via |
|---------|---------|--------------|
| `/posts/welcome-to-my-new-blog/` | `/posts/2026/02/welcome-to-my-new-blog/` | `aliases` |
| `/posts/attractor-dot-pipeline-orchestration/` | `/posts/2026/03/attractor-dot-pipeline-orchestration/` | `aliases` |
| ... all 20 posts ... | `/posts/YYYY/MM/slug/` | `aliases` |

### Hugo Configuration Change

Add to `hugo.toml`:
```toml
[permalinks]
  posts = "/posts/:year/:month/:slug/"
```

This makes Hugo generate URLs at `/posts/2026/02/welcome-to-my-new-blog/` automatically
from the post's `date` frontmatter field, without needing to manually encode the date
in the content path. (The directory structure provides human/filesystem organization;
Hugo's permalink pattern governs the actual URL.)

### Image Reference Changes

**Before** (absolute static path):
```
{{< figure-float src="/images/posts/robot-writing-code.webp" alt="..." >}}
image = "/images/posts/robot-writing-code.webp"
```

**After** (bundle-relative):
```
{{< figure-float src="robot-writing-code.webp" alt="..." >}}
image = "robot-writing-code.webp"
```

The `figure-float` and `figure-block` shortcodes need a one-line update to use
`.Page.Resources.GetMatch` when the `src` doesn't start with `/`:

```gotemplate
{{- $src := .Get "src" -}}
{{- if not (hasPrefix $src "/") -}}
  {{- $res := .Page.Resources.GetMatch $src -}}
  {{- if $res }}{{- $src = $res.RelPermalink -}}{{ end -}}
{{- end -}}
```

The og:image `<meta>` tag in the theme's `<head>` template needs the same treatment
for the `image` frontmatter field.

### `optimize-images.sh` Changes

The script currently hard-codes `IMAGE_DIR="static/images"`. Add a second scan
target for content post bundles:

```bash
CONTENT_POSTS_DIR="content/posts"
```

Adjust the main `find` loop to include both `static/images` and `content/posts/**`
(excluding `thumbs/` and non-image files as it does now).

The `thumbs/` logic currently only triggers for `static/images/posts` and
`static/images/avatars`. Update to trigger for any directory under `content/posts/`
as well (at the same `THUMB_WIDTH_POSTS` setting).

### `generate-post-image.py` Changes

Update to accept `--bundle-dir` argument (or auto-detect from `--post-slug` + `--date`)
and write the output image to the bundle directory instead of `static/images/posts/`.

## Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| og:image meta tag breaks (blank or wrong URL) | Medium | High (social sharing) | Test og:image rendering with a social preview tool after migration |
| Some images shared between posts (would need to stay in static/) | Low | Medium | Audit all image references before moving — keep shared images in `static/images/posts/shared/` |
| RSS feed image URLs break | Medium | Medium | Check RSS output after migration; aliases don't cover RSS entries with absolute URLs |
| `figure-block`/`figure-float` shortcode change breaks existing absolute-path references on other pages | Low | Low | Keep backward compat: only invoke resource lookup for non-`/` prefixed paths |
| CI build time increases | Low | Low | Hugo page resource processing is fast; no impact expected |

## Security Considerations

- No new inputs, APIs, or trust boundaries introduced
- Aliases generate static HTML with `<meta http-equiv="refresh">` — no JS, no injection surface
- Image paths move but are still served as static files — no server-side processing change

## Dependencies

- None (self-contained content restructuring)

## Open Questions

1. **Image referencing style**: Page-resource relative paths (requires shortcode updates)
   vs. updated absolute paths like `/posts/2026/02/slug/image.webp`? Relative is
   cleaner and takes full advantage of leaf bundles; absolute avoids theme changes but
   adds verbose paths to all frontmatter/shortcodes.

2. **Thumbs inside bundles**: Should `thumbs/` subdirectories continue to exist inside
   each post bundle, or should Hugo's built-in image processing (`.Resize "400x"`)
   replace the shell-generated thumbs?

3. **Shared images**: The attractor post uses `software-factory.webp` which could
   plausibly be used by other posts. Should there be a `static/images/posts/shared/`
   bucket for truly shared images, or should each post have its own copy?
